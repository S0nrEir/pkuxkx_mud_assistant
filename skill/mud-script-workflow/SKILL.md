# MUD 脚本工作流

在编写或修改 MUD 助手自动化脚本时使用本 skill。脚本必须以真实服务器消息为依据，而不是只按理想化文本编写。

## 必做流程

1. 先理解现有结构，再动代码。
   - 查看 `scripts/index.json`，确认脚本 id、脚本文件路径和当前描述。
   - 阅读目标脚本文件，通常在 `scripts/` 目录下。
   - 如果需要运行时能力，阅读 `script_system.py`。
   - 如果需要理解消息来源、分发时机、脚本和触发器关系，阅读 `mud_session.py` 中相关流程。

2. 有真实运行日志时，必须参考真实运行日志。
   - 优先查看 `logs/` 中当天或最近会话的 raw/decoded 日志。
   - 从日志中提取触发文本、状态变化文本、奖励文本、失败提示、忙碌提示、命令回显和 NPC 参数。
   - 处理 ANSI/MXP/control sequence 时，优先使用仓库已有清理工具或机器人提供的 `tools.clean()`。
   - 除非用户明确要求，不要改本地存储数据；优先修改解析、过滤、触发、脚本或运行时逻辑。

3. 实现要收窄。
   - 只写与机器人当前任务直接相关的逻辑。不要顺手加入用户没要求的功能：例如不碰网络连接、重连、心跳、配置管理、UI、日志归档等与「机器人要做什么」无关的东西，除非用户明确要求或现有脚本本来就有。
   - 沿用仓库已有脚本、触发器和运行时风格。
   - 状态要显式记录：当前任务、当前步骤、是否等待服务器确认、是否有延迟任务、重试原因、重置条件。
   - 需要延迟行为时，优先加或复用脚本工具函数，例如 `await tools.delay(seconds)`，不要在无关位置散落 sleep。
   - 如果脚本会创建后台延迟任务，必须提供清理逻辑，保证停用脚本或切换脚本后不会继续发送旧命令。

4. 每次创建或修改脚本，都要同步更新脚本元数据。
   - 修改 `scripts/index.json` 中对应脚本的描述字段。
   - 描述必须写清：脚本的作用、主要特点或保护逻辑、触发方式。
   - 脚本已经变成功能脚本后，不要继续保留“示例脚本”这类泛泛描述。

5. 完成后必须基于日志构造测试。
   - 至少对修改过的 Python 文件做语法解析或编译检查。
   - 如果日志中能找到相关参考，必须用日志里的真实文本编写几组模拟消息。
   - 测试要覆盖多轮和不同情况，不只测正常路径。
   - 优先覆盖：失败/重试提示、重复奖励提示、延迟奖励提示、进入/离开状态、缺少前置条件、ANSI/MXP 包裹文本、同一条消息包含多个关键提示。
   - 如果测试发现问题，修复后重新跑相关测试。

6. 测完后自己审查一遍。
   - 检查延迟任务竞态、停用/重载后的旧状态、重复调度、消息顺序变化、同一消息多提示等问题。
   - 确认没有回退用户已有改动。

## 脚本编写指南

- 脚本入口保持为 `async def handle_message(message, tools)`。
- 一进入入口先用 `tools.clean(message)` 得到清理后的文本；后续匹配尽量基于清理文本。
- 对需要长期记忆的信息使用模块级 `state`，例如当前任务参数、路线表、当前步骤、是否在场景中、是否等待确认。
- 对 NPC 给出的动态参数先记录，再行动；例如卦象、目标 NPC、目标地点、任务物品。
- 对路线、阶段、命令序列使用明确表结构，不要把复杂逻辑散在多个字符串判断里。
- 发送命令后要等待服务器确认再推进状态；不要只靠固定时间假设命令成功。
- 遇到“动作还没有完成”“不能移动”“上一个动作还没有完成”等失败提示时，不推进步骤，延迟后重发同一步。
- 遇到完成、奖励、回到起点等提示时，按任务语义重置或结束；如果这些提示会连续出现，要防止重复触发。
- 遇到离开任务区域、脚本停用或切换脚本时，取消未执行的延迟任务并清理等待状态。
- 判断顺序要谨慎：离开、失败、成功、完成这类状态变更通常应优先于房间标题或泛匹配，避免一条消息里多个提示时提前 return。
- 日志里出现的 ANSI、MXP、颜色码、提示符、半行消息都要纳入测试样本。
- 脚本日志使用 `tools.log()` 记录关键状态变化，方便下一次根据运行日志排查。

## 桃花阵脚本模式

当前桃花阵脚本来自真实日志：陆乘风给出卦象，`look tu` 打印八卦方向表，进入桃花阵后出现房间标题和描述，成功移动会出现“充满自信”的提示，移动太快会出现“动作还没有完成，不能移动”，完成一圈后回到起点并获得经验/潜能。

类似迷宫或循环任务脚本应遵循：

- 记录 NPC 给出的任务参数，例如陆乘风算出的卦象。
- 记录或默认日志中观察到的路线/方向表。
- 只在角色进入对应区域后启动自动化。
- 发送当前方向命令后等待服务器确认，再推进到下一步。
- 服务器提示动作未完成或不能移动时，不推进路线，延迟后重试同一命令。
- 完成提示或奖励提示视为一轮结束，从路线开头重新开始。
- 防止“回到起点”“获得经验”“获得潜能”等连续提示导致重复调度。
- 停用脚本或离开区域时停止自动行走并取消待发送命令。

## 完成汇报

最终回复必须说明：

- 改了哪些文件。
- 参考了哪些运行日志或日志片段。
- 跑了哪些测试，覆盖了哪些边界情况。
- 是否同步更新了 `scripts/index.json` 中对应脚本描述。
